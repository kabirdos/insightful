How They Use Claude Code ▼
Craig demonstrates an architect-level approach to Claude Code usage, treating sessions as collaborative work blocks rather than simple Q&A interactions. Sessions average 41 minutes with a clear pattern of front-loading context via file reads before requesting changes. The 1:6 human-to-Claude message ratio reflects a "set direction, then delegate" workflow that maximizes Claude's autonomy while maintaining strategic oversight.

Key pattern: Craig consistently uses /commit and /review-pr skills to bookend coding sessions, creating a structured workflow loop of implement-review-commit that keeps work incremental and reviewable.
Impressive Workflows ▼

Parallel Agent Orchestration

Uses Agent tool to fan-out independent tasks across 3-4 sub-agents, then merges results. Particularly effective for code review and test generation.

UX Mockup Pipeline

Custom /ux-mockup skill generates self-contained HTML mockups with built-in feedback collection, eliminating the Figma-to-code gap entirely.

Safety-Gated Bash

Pre-tool hooks intercept destructive commands (rm -rf, git push --force) before execution, adding a human confirmation step for high-risk operations.

Session Handoff Protocol

/handoff skill captures session state, decisions, and remaining work at end of each conversation, enabling seamless context transfer between sessions.

Tool Mastery ▼
Tool usage shows a mature, deliberate pattern. The Read-before-Edit ratio (1.5:1) indicates Craig rarely makes blind edits, always reading target files first. Grep and Glob are used strategically for codebase exploration rather than brute-force searching. Bash usage is high but well-gated through safety hooks.
Growth Trajectory ▼
Over the 30-day period, session complexity has steadily increased while error rate decreased from 5.1% to 3.2%. Token efficiency improved 18% as Craig refined prompts and adopted more targeted tool invocations. The addition of 3 new custom skills during this period shows active harness development.

v3 Design Note — Standard Report Needs Deeper Analysis:
The standard report currently feels thin compared to the harness report. The next step is to parse a real /insights report and catalog every extractable data point. Until we know exactly what fields are available in the standard /insights output (beyond the current chartData fields like requestTypes, sessionTypes, msgsPerDay, and the narrative sections), we cannot design a richer standard experience. This is a research task, not a design task — we need to run /insights on a real project, capture the full JSON/text output, and document every available data point before iterating on the standard report layout.
