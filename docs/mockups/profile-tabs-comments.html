2 · Alt tab groupings

Three grouping options. Pick one.

Option A — Mirror harness HTML exactly
3 tabs
  • 📊 Dashboard (hero + all charts + tools)
  • 📝 Write-up
  • ✨ Claude Insights

Simplest. Exactly matches user's harness report tabs. Dashboard tab is still long but ~50% of current.

Option B — Recommended
5 tabs
  • 📊 Dashboard (hero + heatmap + How I Work + workflow)
  • 🧰 Skills & Setup (skills, plugins, MCP, hooks, perms)
  • ⚡ Activity (tool usage, langs, git, agents)
  • 📝 Write-up
  • ✨ Claude Insights

Separates what's installed from what was used. Each tab fits one scroll. Best skimmability.

Option C — Skill-forward
4 tabs
  • 📊 Overview
  • 🧰 Toolkit (skills, plugins, MCP, hooks, CLI, agents)
  • 📝 Write-up
  • ✨ Insights

Bets that visitors mostly want to see "what's in this person's toolbox." Fold Activity back into Overview.

3 · Upvote placement

Two layers of voting signal. We already have VoteButton.tsx + voteCounts in the data model — they're stubbed but not displayed.

Layer 1 — Report-level upvote: A single "▲ 24 useful" pill next to the share button / in the upvote-bar under the tabs. Drives the /top leaderboard.
Layer 2 — Per-section upvote: Small ▲ button top-right of each doc-section (see Activity Heatmap above with "▲ 8"). Matches what SectionRenderer already expects (voteCount + voted props). Lets readers signal which parts of a report were useful — valuable data for Claude Insights tuning.
Ship order if time-constrained: Layer 1 first (simple pill, drives /top). Defer Layer 2 until after launch — per-section voting needs moderation thinking (what if someone spams ▲ on the Friction section to mock the author?).
4 · Libraries for inline/anchored comments

You asked whether libraries make Google-Docs-style anchored comments easy. Short answer: no turnkey React library nails this. Here's what's out there.

Option Fit Verdict
Build it yourself
Prisma table + right rail + section IDs 		★★★★★ Schema already has annotations[] keyed by sectionKey. Add Comment table with sectionKey, parentId, body, authorId. Render right-rail. 1–2 days of work, full control.
Giscus
GitHub Discussions-backed 		★★☆☆☆ One thread per page only. Can't anchor per-section without hacks. Also requires a GitHub account to comment → wrong audience for a general public profile page.
Hypothes.is
W3C web annotations 		★★☆☆☆ Overlay sidebar, text-range anchoring. Overkill — their model is annotating arbitrary text; ours is commenting on structured sections. Brand lives in their UI, not yours.
Disqus / Commento / Remark42 ★★☆☆☆ Page-level thread only. Same limitation as Giscus. Also brings tracking (Disqus) or self-hosting overhead.
Liveblocks · Threads
liveblocks.io/docs/products/comments 		★★★★☆ Purpose-built for Figma-style anchored threads. Real-time out of the box. But: hosted service with usage pricing, adds dependency, and their data (not yours). Good if we want real-time multi-user co-presence later; overkill for v1.
react-comments-section / @chatscope ★★☆☆☆ UI kits only — they render the thread tree but you own the data. Saves maybe 4 hours of component work. Not worth the dependency for what is, structurally, a simple form + list.
Recommendation: build it yourself. The right rail above is ~200 lines of React + a Comment Prisma model. You already have auth, Prisma, and section keys. Using a library to save half a day now costs you lock-in and a third-party UI that will drift from your design system. Revisit Liveblocks only if real-time co-presence becomes a feature ask.
Decision (2026-04-12): Deferred. Keep the existing bottom-of-page CommentSection as-is for launch. Revisit anchored/rail comments post-launch.
5 · Open questions before building
  1. Mobile comments. Right rail collapses below content on mobile. Acceptable, or do we need a floating "💬 3" button that opens a drawer?
  2. Comment ownership. Can the report author delete others' comments on their profile, like Medium?
  3. Anonymous comments? Probably no — require login, mirrors upvote auth.
  4. Notifications. Email the author when someone comments? (Resend is already on the stack.)
  5. Per-tab comment badges. Should each tab show an unread-count badge (like the red dot on Activity in the mock)? Probably yes, low-cost, high-value.
  6. Cross-tab thread visibility. Decided 2026-04-12: comments stay at the bottom of the page (not in a rail) and show all threads across tabs, with an anchor label on each. See v2.
